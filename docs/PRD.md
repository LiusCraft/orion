## 1. 简介

### 1.1. 项目概述
**Orion** 是一款面向研发和运维团队的智能运维保障产品。它旨在解决在**发布、配置变更**等高风险操作后，需要人工持续盯守指标看板和告警列表的痛点。Orion 通过自动化监控、智能分析、预案执行和主动通知，将人员从被动的、重复性的观察工作中解放出来，实现问题的**自动发现、自动诊断、自动决策和自动恢复**，从而提升系统稳定性，加速变更流程，并减少人为失误。

### 1.2. 问题陈述
当前，在进行线上发布或重大变更时，工程师通常需要：
1.  在多套监控系统（如 Prometheus、APM）、日志系统（如 ELK）和告警平台间不断切换。
2.  精神高度集中，人工判断指标波动是正常现象还是故障前兆。
3.  一旦发现告警，需要手动下钻分析，定位根因，过程耗时且对经验依赖高。
4.  即使发现问题，执行回滚或止损操作也可能存在延迟，导致影响扩大。
这个过程效率低下，消耗大量人力，且在深夜等时段容易因疲劳导致响应不及时。

### 1.3. 目标与范围
**核心目标：** 为每一次关键变更提供“自动驾驶”般的智能保障，实现“无人值守”的可靠发布与变更。
**范围：**
*   集成各类监控和数据源。
*   与发布系统、变更管理系统对接。
*   提供灵活的任务编排和指标观察配置。
*   实现多通道、可升级的通知机制。
*   提供自动化诊断分析和根因报告。
*   支持与应急预案工具集成，实现自动决策和执行。

## 2. 用户角色与 persona

| 角色 | 主要目标 | 使用场景 |
| :--- | :--- | :--- |
| **研发工程师** | 安全、高效地完成应用发布，无需熬夜盯屏。 | 在发布系统启动发布，并同时启动一个 Orion 保障任务。 |
| **运维工程师** | 确保基础设施或全局配置变更的稳定性，快速响应异常。 | 在执行数据库扩容或网络配置变更后，创建观察任务。 |
| **SRE/稳定性负责人** | 减少人为操作失误，提升系统整体 SLA，沉淀应急经验。 | 配置和维护平台级的监控指标、分析规则和应急预案库。 |

## 3. 功能需求

### 3.1. 核心概念：保障任务
一次保障任务（Guard Duty）是 Orion 的核心执行单元。用户在执行变更（如发布）时创建，它定义了：
*   **监控对象：** 要观察的应用、服务或基础设施。
*   **观察窗口：** 任务执行的时间范围（如：发布开始后的60分钟）。
*   **监控指标与告警规则：** 需要关注的具体指标和阈值。
*   **关联动作：** 发现问题后执行的操作（如分析、通知、执行预案）。
*   **升级策略：** 通知频次和升级条件（如：5分钟未确认则电话通知）。

### 3.2. 功能详述

#### 3.2.1. 任务管理
*   **FR1.1 创建任务：** 用户可通过 Web UI 或 API 创建保障任务。任务可与发布单ID、变更单ID关联。
*   **FR1.2 任务模板：** 支持创建任务模板（如“Java应用发布模板”），预置常用的指标和规则，方便快速使用。
*   **FR1.3 任务列表与详情：** 可查看所有历史任务和当前任务的实时状态（“运行中”、“已结束”、“已触发动作”）。

#### 3.2.2. 数据与集成
*   **FR2.1 监控数据源集成：** 支持接入主流监控工具（如 Prometheus, Thanos, VictoriaMetrics, Datadog）。
*   **FR2.2 告警平台集成：** 支持从告警平台（如 Alertmanager, Grafana, Zabbix）主动拉取或接收webhook推送的告警事件。
*   **FR2.3 变更系统集成：** 支持与发布系统、CI/CD流水线、变更管理系统对接，自动触发和结束保障任务。

#### 3.2.3. 监控与观察
*   **FR3.1 多指标监控：** 用户可配置一组需要观察的指标（如：QPS、错误率、响应时间、JVM内存使用率）。
*   **FR3.2 智能基线：** 支持基于历史数据（如前7天同时段数据）自动计算动态基线，而非固定阈值，减少误报。
*   **FR3.3 关联性分析配置：** 可配置简单的关联规则（如：**如果**错误率飙升 **并且** 响应时间变长 **并且** 部署版本变化，**则** 很可能是一次坏发布）。

#### 3.2.4. 告警与通知
*   **FR4.1 多通道通知：** 支持通过钉钉/飞书群消息、邮件、短信、电话（语音）等方式发送告警。
*   **FR4.2 通知内容：** 消息须包含关键信息：任务名、问题摘要、触发指标、关联变更ID、诊断报告链接。
*   **FR4.3 升级策略：** 支持配置通知升级规则（例：第一次告警发到群；10分钟后若未恢复也未确认，则短信通知负责人；再5分钟后则自动打电话）。

#### 3.2.5. 智能诊断与分析
*   **FR5.1 自动下钻分析：** 当触发告警后，系统自动关联查询相关指标、日志和链路追踪数据。
*   **FR5.2 根因报告生成：** 自动生成一份诊断报告，内容包括：
    *   异常时间点
    *   核心异常指标（附图表）
    *   可能的原因（如：**90%** 的概率与刚发生的发布有关）
    *   关联的变更信息
    *   建议的解决方案（如：执行回滚、扩容、重启服务）

#### 3.2.6. 自动化动作
*   **FR6.1 手动审批动作：** 用户可配置“建议动作”，需用户点击确认后执行（如：“检测到问题，是否执行回滚？”）。
*   **FR6.2 自动执行预案：** 对于高信任度的场景，用户可授权平台在满足特定条件（如：触发P0告警且15分钟内无人响应）时，自动执行预设的应急预案（通过对接运维工具或API），如：
    *   调用发布系统的回滚API
    *   调用扩容接口
    *   执行某个预定的脚本

#### 3.2.7. 仪表盘与报告
*   **FR7.1 实时状态仪表盘：** 展示当前所有运行中任务的状态，全局视图。
*   **FR7.2 历史报告：** 所有保障任务的执行详情、分析报告和操作记录均可回溯审计。

## 4. 非功能需求

*   **NF1 可靠性：** 平台自身必须高可用，不能成为单点故障。保障任务的执行成功率需 > 99.9%。
*   **NF2 性能：** 从指标异常被捕捉到发出通知，延迟应小于60秒。数据分析过程应在3分钟内完成。
*   **NF3 安全性：** 所有API调用和集成配置需进行认证和授权。执行自动化动作需有严格的权限控制和操作审计。
*   **NF4 可扩展性：** 架构设计应能支持同时运行上千个保障任务，并能方便地接入新的数据源和动作执行器。

## 5. 未来迭代考虑

*   **预测性分析：** 在问题发生前，基于趋势预测潜在风险。
*   **机器学习驱动：** 使用机器学习模型进行更复杂的异常检测和根因分析，减少误报。
*   **更丰富的动作库：** 预置更多开箱即用的应急预案动作。
*   **成本优化洞察：** 在保障稳定性的同时，观察资源利用率，给出成本优化建议。
